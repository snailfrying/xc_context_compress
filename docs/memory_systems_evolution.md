# 长期记忆系统演进与选型

> **定位**：第二节"会话压缩与长期记忆"的补充阅读材料
> 从记忆技术的七代演进出发，介绍当前市场主流框架，阐述 Context Distiller 为何选择 **OpenClaw + Mem0** 双后端方案。

---

## 一、为什么 Agent 需要长期记忆？

大语言模型本质上是**无状态**的——每次调用独立处理，不记得上一轮对话说了什么。为了让 Agent 具备连续性和个性化能力，需要在模型之外构建记忆系统。

记忆系统要解决三个核心问题：

| 问题 | 说明 |
|------|------|
| **存什么** | 从海量对话中提取值得持久化的知识（事实、偏好、规则） |
| **怎么存** | 选择什么数据结构和存储介质（向量、关键词索引、图谱） |
| **怎么找** | 在需要时高效召回最相关的记忆，注入上下文 |

这三个问题的答案随着技术演进不断变化，形成了七代记忆技术的迭代。

---

## 二、七代记忆技术演进 (1960s → 2026)

### 第一代：符号知识库 (1960s–1980s)

以 MYCIN 专家系统、Cyc 常识知识库为代表。知识由人工编写为 IF-THEN 规则或逻辑断言，通过前向链/后向链推理引擎使用。

- **优势**：推理路径可解释
- **瓶颈**：知识获取成本极高（Cyc 耗资数亿美元），规则冲突时系统脆性崩溃，无法处理不确定性

### 第二代：关系数据库 (1990s–2000s)

SQL 数据库（MySQL、PostgreSQL）用表格存储用户信息、会话记录、消息历史，通过 JOIN 和索引实现结构化查询。

- **优势**：ACID 事务保证一致性，查询语言成熟
- **瓶颈**：Schema 刚性（改表需迁移），**无语义理解**——只能精确匹配，"苹果手机"和"iPhone"无法关联

### 第三代：NoSQL + 全文检索 (2000s–2010s)

MongoDB（文档存储）、Redis（内存缓存）、Elasticsearch（全文检索）。Schema-less 设计支持灵活字段，BM25 算法支持关键词相关性排序。

- **优势**：水平扩展好，全文检索能力强
- **瓶颈**：仍然停留在**词汇匹配**层面，无法理解语义相似性

### 第四代：向量数据库 (2018–2023)

Pinecone、Weaviate、Milvus、Qdrant、Chroma 等。将文本通过 Embedding 模型映射为高维向量，通过 HNSW 等近似最近邻算法实现**语义检索**。

- **优势**：首次实现"找语义相近的记忆"，支持多模态（文本、图像、音频）
- **瓶颈**：纯向量检索在**精确匹配**场景表现差——搜索"订单号 12345"时，向量化数字效果不佳；时间范围过滤和元数据筛选效率低

### 第五代：混合检索 (2023–2024)

将向量检索（语义相似）与关键词检索（精确匹配）融合：

```
最终得分 = α × 向量相似度 + (1-α) × 关键词匹配度
```

Weaviate 官方测试数据显示：纯向量检索 Recall@10 = 0.72，纯 BM25 = 0.68，**混合检索 = 0.89**（提升 24%）。

| 查询场景 | 纯向量 | 混合检索 |
|---------|--------|---------|
| "帮我找上次讨论的 API 设计" | 语义匹配可用 | 语义 + 关键词"API"双路命中 |
| "订单号 12345 的问题" | 数字向量化差 | FTS 精确匹配 |
| "最近 7 天的对话" | 需后过滤 | SQL WHERE 直接筛选 |

### 第六代：智能体记忆框架 (2023–2025)

LLM 的爆发催生了专门为 Agent 设计的记忆框架。核心转变：**从"开发者手动管理记忆"到"LLM 自动提取和维护记忆"**。

这一代的标志性事件：

| 时间 | 事件 | 意义 |
|------|------|------|
| 2023.04 | Auto-GPT 发布 | 首次展示 Agent 持久记忆的可能性 |
| 2023.04 | 斯坦福"生成式智能体"论文 | 提出记忆的存储、反思、检索三阶段架构 |
| 2023.10 | MemGPT 论文发表 | 受操作系统启发，LLM 像管理虚拟内存一样管理自身上下文 |
| 2024.01 | Mem0 发布 | 第一个"记忆即服务"产品，自动提取 + 智能去重 |
| 2024.09 | Letta（MemGPT 商业化）融资 $10M | 验证了记忆系统的商业价值 |

代表框架涌现：LangChain Memory、LlamaIndex Memory、Mem0、Zep、AutoGen 等。

### 第七代：架构分化与互补方案 (2025–2026)

第六代框架逐渐成熟后，市场并未收敛为单一形态，而是沿**架构设计理念**分化为两条路线。注意：这里的区别不是"本地 vs 云端"（两者都可以本地或云端部署），而是**记忆管理的控制权归属**不同。

```
┌───────────────────────────────────────────────────────┐
│                Agent 记忆系统架构分化                      │
├──────────────────────┬────────────────────────────────┤
│   显式管理架构          │       隐式管理架构               │
│   (开发者/用户控制)      │       (LLM 自动驱动)            │
│                        │                               │
│   代表: OpenClaw        │   代表: Mem0                   │
│                        │                               │
│   存什么 → 业务层决定    │   存什么 → LLM 自动判断          │
│   怎么存 → 透明可编辑    │   怎么存 → 框架内部处理          │
│   怎么找 → 混合检索      │   怎么找 → 语义检索 + 图谱       │
│                        │                               │
│   核心价值: 可控、可审计  │   核心价值: 自动化、智能化        │
└──────────────────────┴────────────────────────────────┘
```

两种架构并非替代关系，而是**互补关系**——这正是 Context Distiller 采用双后端架构的出发点。

## 三、当前市场主流框架全景

2024–2026 年间，Agent 记忆领域涌现出大量框架。按**架构设计理念和业务定位**可分为四类：

### 3.1 通用 Agent 框架的内置记忆模块

这类框架的核心是 Agent 编排，记忆只是附属功能。

| 框架 | 记忆方式 | 特点 | 局限 |
|------|---------|------|------|
| **LangChain Memory** | Buffer / Summary / VectorStore 等多种模块可选 | 灵活、可组合、生态最大 | 是积木而非成品，需自行搭建提取/去重/检索全流程 |
| **LlamaIndex Memory** | ChatMemoryBuffer + VectorMemory 组合 | 与文档检索（RAG）紧密集成 | 侧重文档场景，对话记忆功能较基础 |
| **AutoGen** | 消息列表作为记忆载体 | 多 Agent 群聊记忆共享简单直接 | 无持久化机制，会话结束即丢失 |
| **CrewAI** | Entity Memory + Contextual Memory + User Memory | 角色化设计，记忆按职责分类 | 绑定 CrewAI 生态，通用性有限 |

**小结**：这些框架提供的是记忆"积木"，适合有能力自行搭建的团队。但对于需要**开箱即用**的长期记忆方案，仍然不够。

### 3.2 专用记忆服务（隐式管理架构）

这类产品的核心是记忆本身，以独立服务形态接入任意 Agent 框架。架构特点：**LLM 驱动的自动记忆生命周期管理**——提取、去重、更新、检索全部由框架内部完成，开发者只需调用 API。

| 框架 | 架构特色 | 记忆管理方式 | 适用业务场景 |
|------|---------|------------|------------|
| **Mem0** | 两阶段管线（提取→更新），LLM 自动判断记忆的增删改 | 隐式——传入对话即自动完成 | 需要自动化记忆管理的产品、跨会话个性化 |
| **Zep (Graphiti)** | 时序知识图谱引擎，时间作为一等维度 | 隐式——自动构建实体-关系-时间图谱 | 需要时间维度推理（"上个月用户还偏好 X"） |
| **Cognee** | 自进化知识图谱 + 14 种检索模式 | 隐式——ECL 管线自动提取、构图、优化 | 文档密集型、知识图谱为核心的应用 |

> 注：这些框架均支持云端和本地自托管部署。"隐式管理"指的是架构设计，而非部署方式。

### 3.3 Agent 运行时（记忆深度耦合架构）

记忆不是独立模块，而是与 Agent 执行引擎深度耦合。架构特点：**LLM 自身作为记忆管理者**——Agent 通过函数调用决定存什么、删什么、改什么。

| 框架 | 架构理念 | 记忆管理方式 | 局限 |
|------|---------|------------|------|
| **Letta (MemGPT)** | 受操作系统启发，LLM 管理自身的"虚拟内存" | 自编辑——Agent 自主发起记忆的读写删改 | 是完整运行时，需替换整个 Agent 栈 |

### 3.4 显式管理架构

架构特点：**开发者/用户完全控制记忆的生命周期**——存什么、怎么分类、何时删除，全部由业务层显式决定。记忆内容透明可读，存储结构可直接审计。

| 框架 | 架构特色 | 记忆管理方式 | 局限 |
|------|---------|------------|------|
| **OpenClaw** | SQLite + FTS5 + 向量索引，记忆以 Markdown 可读存储 | 显式——业务层调用 store/update/forget | 无自动提取能力，需业务层决定"存什么" |

> 注：OpenClaw 同样支持本地或服务器部署。"显式管理"是其架构设计选择，不等于"仅限本地"。

## 四、为什么选择 OpenClaw + Mem0？

面对众多选择，Context Distiller 选择这两个框架作为双后端，基于以下分析：

### 4.1 排除法：为什么不选其他框架？

| 候选方案 | 不选的原因 |
|---------|-----------|
| **LangChain / LlamaIndex 内置记忆** | 是零件不是成品。需要自行实现提取逻辑、去重机制、检索策略、持久化层。等于从零搭建记忆系统 |
| **Zep** | 图谱构建代价高昂（单次对话超 60 万 tokens），社区版已停止维护，推向付费云端。对于不需要时序推理的场景是过度设计 |
| **Letta (MemGPT)** | 是完整的 Agent 运行时，不是可插拔的记忆层。选择 Letta 意味着放弃现有的 Agent 架构（Ollama + FastAPI），改造成本过高 |
| **Cognee** | 面向文档/知识图谱场景优化，对于对话式记忆偏重。部署和配置复杂度高 |
| **CrewAI / AutoGen 内置** | 绑定特定框架生态，无法独立使用。记忆功能相对基础 |

### 4.2 架构对比：两种记忆管理哲学

OpenClaw 和 Mem0 的核心差异不在于"本地 vs 云端"（两者都可以本地或云端部署），而在于**架构设计理念**根本不同，由此带来不同的业务适配性。

#### OpenClaw：显式管理架构

设计哲学——**"记忆的控制权在开发者手中"**。

| 架构特征 | 说明 |
|---------|------|
| **存储透明** | 记忆以 Markdown 文件 + SQLite 存储，人类可直接阅读、编辑、删除 |
| **检索可控** | FTS5（关键词精确匹配）+ 向量（语义相似）双路混合检索，权重可调 |
| **生命周期显式** | 业务层决定存什么、何时更新、何时删除，没有"黑盒"自动行为 |
| **依赖极简** | 单个 SQLite 文件承载全部能力，无需额外服务 |

**架构优势**：可审计、可调试、确定性强。你知道记忆里存了什么、为什么存、怎么检索到的。

**架构局限**：没有自动提取能力——"从对话中识别什么值得记住"需要业务层自行实现。记忆质量完全取决于上层调用逻辑。

#### Mem0：隐式管理架构

设计哲学——**"记忆的管理交给 LLM"**。

| 架构特征 | 说明 |
|---------|------|
| **自动提取** | 传入对话文本，LLM 自动判断哪些信息值得持久化，无需人工标注 |
| **语义去重** | "用户喜欢咖啡"和"用户爱喝拿铁"会被识别为语义相关，合并而非重复存储 |
| **冲突检测** | 旧记忆"用户擅长 Python"遇到新信息"用户转向 Go"时，自动判断应更新而非新增 |
| **可选图谱** | 支持接入 Neo4j 构建实体关系图谱，实现多跳关系推理 |

**架构优势**：自动化程度高，开发者无需设计记忆提取规则，"开箱即用"的智能记忆管理。

**架构局限**：提取过程依赖 LLM 推理（有额外成本），提取逻辑不完全透明（无法精确控制"它觉得什么值得记"），确定性较低。

#### 架构差异导致的业务适配差异

| 业务维度 | OpenClaw（显式管理） | Mem0（隐式管理） |
|---------|-------------------|-----------------|
| **记忆写入** | 开发者编写规则决定"存什么" | LLM 自动判断"存什么" |
| **记忆更新** | 显式调用 update，完全可预测 | 自动检测冲突并合并，行为由 LLM 决定 |
| **记忆检索** | 混合检索（FTS + 向量），权重可调 | 语义检索为主，可选图谱增强 |
| **可审计性** | 高——Markdown 可读，SQL 可查 | 中——需通过 API 查看，提取逻辑不透明 |
| **自动化程度** | 低——需业务层驱动 | 高——传入对话即完成全流程 |
| **推理成本** | 仅检索时需要 Embedding | 每次存储都需要 LLM 推理 |
| **确定性** | 高——相同输入产生相同结果 | 低——LLM 提取结果可能不一致 |

### 4.3 双后端互补：为什么需要两种架构？

两种架构解决的是**记忆管理光谱的两端**：

```
完全可控                                              完全自动
  ◄──────────────────────────────────────────────────────►
  │                                                      │
  OpenClaw                                            Mem0
  显式管理                                            隐式管理
  │                                                      │
  适合:                                               适合:
  · 需要精确控制记忆内容                               · 需要自动化记忆管理
  · 合规审计要求高                                     · 对话量大，人工管理不现实
  · 记忆逻辑已有明确业务规则                            · 需要智能去重和冲突检测
  · 对确定性要求高                                     · 快速上线，不想自建提取逻辑
```

Context Distiller 将两者封装在统一接口之后，业务层通过配置切换，无需修改代码：

```
┌──────────────────────────────────────────────────────────┐
│               Context Distiller 双后端架构                 │
│                                                          │
│   OpenClaw (显式管理)            Mem0 (隐式管理)           │
│   ┌──────────────┐            ┌──────────────┐           │
│   │ 业务层决定存什么 │            │ LLM 自动决定   │           │
│   │ 混合检索可调参  │    互补     │ 语义去重+冲突   │           │
│   │ Markdown 可审计│  ◄──────►  │ 图谱可选       │           │
│   │ 依赖极简       │            │ 框架无关       │           │
│   └──────────────┘            └──────────────┘           │
│          ↓                           ↓                   │
│     统一接口: search / store / update / forget            │
│     配置一键切换，业务层代码无需修改                         │
└──────────────────────────────────────────────────────────┘
```

**选型建议**：

| 业务场景 | 推荐后端 | 核心理由 |
|---------|---------|---------|
| 合规审计要求高、需要精确控制记忆内容 | OpenClaw | 显式管理，记忆内容完全可审计 |
| 记忆逻辑已有明确业务规则 | OpenClaw | 确定性高，规则可编码 |
| 对话量大、人工管理记忆不现实 | Mem0 | 自动提取免去人工标注 |
| 快速上线、不想自建提取/去重逻辑 | Mem0 | 开箱即用的智能记忆管理 |
| 部分记忆需精确控制 + 部分需自动化 | **混合模式** | 按业务需求将不同类型记忆路由到不同后端 |

## 五、多 Agent 记忆管理的五种架构模式

当系统中存在多个 Agent 协作时，记忆如何在它们之间共享和隔离，是架构设计的关键决策。

| 模式 | 代表框架 | 原理 | 适用场景 |
|------|---------|------|---------|
| **共享状态对象** | LangGraph | 所有 Agent 读写同一个中心状态字典，通过 Reducer 函数合并并发更新 | 精确控制数据流的工作流编排 |
| **角色化记忆池** | CrewAI | 按 Agent 角色分配不同类型的记忆（实体记忆、上下文记忆、用户记忆） | 团队分工明确的业务流程 |
| **消息传递** | AutoGen | 记忆以消息列表形式在 Agent 间传递，群聊管理者负责协调 | 多 Agent 对话决策场景 |
| **外部记忆服务** | Mem0 / Zep | 独立记忆层，Agent 通过 user_id + agent_id 作用域隔离读写 | **框架无关**，任意 Agent 栈可接入 |
| **自管理记忆** | Letta | 每个 Agent 自主管理自身记忆层次（核心记忆、归档记忆），通过 API 实现跨 Agent 共享 | 研究导向的自主记忆实验 |

Context Distiller 采用的是**"外部记忆服务"**模式——通过 `(user_id, agent_id)` 复合键实现多租户隔离，不同 Agent 的记忆互相独立，同一用户下可跨 Agent 检索。

**行业趋势**：2025–2026 年的主流方向是混合架构——外部记忆服务处理跨 Agent 持久化，各框架内部管理自己的工作记忆。具体选择显式管理（OpenClaw）还是隐式管理（Mem0）作为外部记忆层，取决于业务对可控性和自动化程度的要求。

## 六、记忆技术发展脉络总览

```
1960s        1990s        2000s        2018         2023         2024         2025-26
  │            │            │            │            │            │            │
  ▼            ▼            ▼            ▼            ▼            ▼            ▼
符号规则  →  关系数据库  →  NoSQL     →  向量数据库  →  混合检索  →  Agent记忆  →  架构分化
                          全文检索      语义检索      双路融合      框架       显式+隐式

核心跃迁：
  精确匹配 ────→ 语义理解 ────→ 混合检索 ────→ 自动提取 ────→ 架构互补
```

每一代都在解决上一代的核心瓶颈：符号系统无法扩展，关系数据库缺乏语义理解，向量数据库不擅长精确匹配，混合检索缺少自动化能力。到了第七代，业界认识到**没有单一架构能同时满足"完全可控"和"完全自动"**，因此分化出显式管理和隐式管理两种互补路线。

| 方向 | 趋势 | 预期影响 |

| **联邦学习记忆** | 本地训练 + 云端聚合，差分隐私保护 | 在不暴露原始数据的前提下实现协作学习 |
| **时序知识图谱** | 记忆不仅记录"是什么"，还记录"何时成立、如何演变" | 支持时间维度推理（"上个月用户还偏好 X，现在转向了 Y"） |
| **可移植记忆护照** | 记忆跨平台、跨应用流动的标准协议 | 用户更换 Agent 不丢失积累的个性化知识 |
| **自进化记忆** | 记忆网络自动剪枝过时节点、强化高频连接、推导新事实 | 记忆不再是静态存储，而是持续优化的知识结构 |
